01 · Executive Summary

执行摘要

STEMotion 当前已具备 RAG 学生助学、教师助教、Deep Interaction 可视化生成、本地交互库、模型 Profile 设置、学科 Skill、本地知识库、Hybrid Retriever、多 Agent 质量检查和可视化管线等十项基础能力。但现有架构呈现「多个功能页面拼接」的状态，缺乏面向真实教学场景的课程闭环感。

核心 Takeaways

  • 重构目标：从「功能堆叠」升级为「课程闭环」，建立 知识库 → RAG → Agent 质检 → 结构化教学输出 → Deep Interaction → Artifact → 教学资产库 的完整技术链
  • 产品层拆分为 6 个一级模块：学生学习、教师教学、可视化实验、教学资产、知识库管理、系统设置
  • Agent 体系整理为 5 条流水线，覆盖知识构建、学生 RAG、教师 RAG、实验生成、交付评估
  • 新增知识库健康度管理页面，满足比赛对「专业知识库」和「内容可追溯」的核心要求
  • 建立 RAG 到 Lab 的桥接机制，学生学习或教师备课结果可一键跳转生成交互实验
本文档回答的问题
如何将 STEMotion 从多页面功能拼接重构为面向高校课程的智能教学资产生成平台？重构后的模块划分、Agent 架构和技术桥接如何落地？比赛交付材料如何对齐？
02 · Background

背景与问题定义

XH202620 比赛要求关注真实教学场景、专业知识库、RAG 可追溯、典型案例、用户反馈和可推广性。STEMotion 需要从技术原型升级为可交付的教学产品。

当前现状

STEMotion 已完成的核心能力包括：RAG 学生助学与教师助教模块、Deep Interaction 可视化生成管线、本地交互库与模型 Profile 设置、学科 Skill 与本地知识库、Hybrid Retriever 与多 Agent 质量检查。这些能力覆盖了从知识检索到交互实验生成的完整技术栈。

核心问题

现有架构的主要问题在于各功能模块之间缺乏统一的课程工作流叙事。学生助学、教师助教、可视化实验和交互库各自独立运行，没有形成「从知识到资产」的闭环。对于比赛评审而言，这表现为功能丰富但场景割裂。

重构方向不是「把现有页面再修好看一点」，而是把项目整体升级为：面向高校课程的智能教学资产生成平台。减少「页面堆叠感」，强化「课程闭环感」。

衡量标准

维度 当前水平 目标水平 差距
模块命名 Student / Teacher / Visualization / Interactions Learn / Teach / Lab / Assets / Knowledge / Settings 语义不匹配教学场景
知识库可见性 仅在设置页显示文件数和 chunk 数 独立知识库健康度页面，含覆盖模块、缺失分析、来源可信度 缺少知识库质量管理入口
RAG 到 Lab 桥接 无直接跳转 学习或备课回答完成后可一键生成交互实验 prompt 两个核心能力未打通
Agent 分类 分散在代码中，无统一文档 5 条流水线，文档化已实现 / 部分实现 / 规划中状态 架构文档缺失
比赛材料 无结构化提交目录 完整 competition_submission 目录与 readiness checklist 交付物未对齐
03 · Product Architecture

产品层重构：六大模块

将项目重构成 6 个一级模块，左侧导航对应「学生学习 → 教师教学 → 可视化实验 → 教学资产 → 知识库 → 设置 → 关于」，形成从学习到资产沉淀的完整课程工作流。

模块总览

STEMotion
├── 学生学习 Learn      面向学生，底层用 RAG
├── 教师教学 Teach      面向教师，底层用 RAG
├── 可视化实验 Lab      继承 Deep Interaction
├── 教学资产 Assets     原交互库升级
├── 知识库 Knowledge    新增一级模块
└── 系统设置 Settings   模型 Profile + 系统健康

1. 学生学习 Learn

面向学生，底层复用 RAG 管线。核心任务包括知识讲解、分步解题、错因诊断、学习报告、错题本和推荐练习。输出包含结构化回答、公式推导、单位检查、引用来源、学习建议和可视化提示。

默认学科为 physics_mechanics，默认问题示例为大学物理力学中的斜抛运动建模。新增轻量功能：LearningReportPanel（学习报告面板）、SaveToMistakeBookButton（保存错题本按钮）、GenerateLabFromAnswerButton（从回答生成交互实验按钮）。

2. 教师教学 Teach

面向教师，底层同样复用 RAG。核心任务包括课堂备课、演示设计、练习生成、Rubric 评分标准、教学材料导出和一键生成可视化实验。输出包含教学目标、课堂导入、板书设计、互动问题、演示参数、课后练习、评分标准和引用来源。

新增功能：ExportTeachingMaterialButton（支持 Markdown、剪贴板、PPT 大纲文本、练习题单等导出格式）、RubricGeneratorPanel、GenerateLabPromptFromTeachingPlan。

3. 可视化实验 Lab

继承 Deep Interaction 管线，而非简单 SVG 页面。核心任务包括 Guided Planning、LearningBlueprint、Subject Schema 校验、Verified Template 匹配、HTML/SVG/Canvas artifact 生成、多 Agent 评审、Repair 修复、artifact_ready 状态和保存到教学资产库。

典型案例库：

编号 实验名称 核心交互
1 非零高度斜抛运动交互实验 调节 h、v₀、θ、g，观察轨迹与落地参数
2 匀速圆周运动方向与向心加速度演示 调节角速度与半径，观察速度矢量变化
3 机械能守恒滑块实验 调节斜面角度与摩擦系数，观察能量转化
4 动量守恒碰撞实验 调节质量与初速度，观察弹性 / 非弹性碰撞
5 简谐振动实验 调节弹簧常数与振幅，观察位移与速度曲线

4. 教学资产 Assets

替代原 /interactions 主导航名称。「交互库」改名为「教学资产」，因为前者听起来像开发者功能，而后者更符合比赛叙事和教育场景。

资产类型包括：RAG 学习报告、错题记录、教师备课方案、练习题单、交互实验 artifact、可视化演示。新增字段：subject、module、assetType、tags、source、createdAt、updatedAt、version、qualityLevel。如果已有 artifact store，尽量兼容旧数据，新增字段可选。

5. 知识库 Knowledge

新增的一级模块。不再只在设置页显示「17 文件 / 20 片段」，而是做成可见的知识库管理与健康度页面。这非常符合比赛要求，因为赛题很强调「专业知识库」和「内容可追溯」。

功能包括：学科知识库状态展示、文件数与 chunk 数、覆盖模块与缺失模块、最近更新时间、来源可信度、重新 ingest 指令展示、检索测试入口、知识库健康度评分。展示学科包括 physics_mechanics、advanced_math、chemistry、computer_science。

6. 系统设置 Settings

保留模型 Profile 配置功能，增加系统健康状态展示：当前 active model 是否存在、API Key 脱敏显示、RAG 索引是否存在、physics_mechanics 是否已索引、Deep Interaction 是否可用、localStorage 是否可用、最近一次构建或检查状态。

04 · Agent Architecture

Agent 架构：五条流水线

将当前 Agent 体系整理为 5 条流水线，覆盖从知识库构建到比赛交付的完整链路。代码中暂时没有对应 Agent 的，先在文档中说明为「规划中」，不伪造实现。

1. Knowledge Build Agents

用于构建知识库。包括 SourceScoutAgent（公开资料检索）、SourceComplianceAgent（版权与来源检查）、ContentCleanerAgent（内容清洗）、KnowledgeStructuringAgent（知识库 Markdown 生成）、ChunkIndexAgent（chunks / keyword / vector 索引）、KnowledgeHealthAgent（知识库健康度分析）。

2. Learning RAG Agents

用于学生学习。包括 QueryPlannerAgent、HybridRetrieverAgent、EvidenceAssemblerAgent、AnswerGeneratorAgent、FormulaUnitCheckAgent（公式与单位检查）、MisconceptionDiagnosisAgent（错因诊断）、LearningReportAgent（学习报告）、JudgeAgent、RevisionAgent。

3. Teaching RAG Agents

用于教师教学。包括 TeachingPlannerAgent、LessonDesignAgent、ExerciseGeneratorAgent（练习题单生成）、RubricGeneratorAgent（评分 Rubric 生成）、TeachingExportAgent（教学材料导出）、EvidenceGroundingAgent。

4. Lab Generation Agents

继承 Deep Interaction。包括 GuidedPlanningAgent、LearningDesignAgent、SubjectSchemaValidator、TemplateMatcher、WidgetHtmlAgent、TeacherActionAgent、SafetyEvaluator、RuntimeEvaluator、PedagogyEvaluator、UxEvaluator、JudgeAgent、RepairAgent、QualityReportAgent。重点增强：大学物理力学模板库、RAG → Lab Prompt 桥接、artifact 版本说明。

5. Delivery & Evaluation Agents

用于比赛交付。包括 EvaluationCaseAgent（维护测试案例）、RubricMappingAgent（映射比赛评分维度）、DemoScriptAgent（生成演示脚本）、UserFeedbackAgent（汇总真实用户反馈）、SubmissionPackAgent（检查提交材料完整性）。

Agent 实现状态说明：代码中已实现的 Agent 直接标注「已实现」；部分功能可用的标注「部分实现」；尚未编码的标注「规划中，文档占位」。不伪造任何实现状态。

技术桥接与路由设计

路由重构遵循「不删除旧代码、不破坏已有 API」的工程约束，通过 redirect 或 wrapper 实现新旧路由兼容。RAG 到 Lab 的桥接是本次重构的核心技术亮点。

RAG 到 Lab 桥接

在 /learn 和 /teach 的回答完成后增加「生成交互实验」按钮。点击后：根据当前回答、主题、taskType 生成 lab prompt；跳转到 /lab；将 prompt 填入 Lab 输入框；不自动开始生成，用户仍需确认 Guided Plan。

示例 prompt：

基于刚才的非零初始高度斜抛运动模型，生成一个大学物理交互实验。学生可以调节 h、v₀、θ、g，观察轨迹、落地时间、最大高度、水平射程和速度分量随时间的变化。

07 · Engineering Constraints

工程约束与测试验证

本次重构允许大幅调整前端路由、页面命名、功能分区、Agent 组织方式和文档结构，但必须遵守以下工程约束。

工程红线

  1. 不删除核心旧代码
  2. 不破坏 RAG API
  3. 不破坏 Deep Interaction 管线
  4. 不破坏模型设置
  5. 不删除已有知识库
  6. 不伪造用户反馈
  7. 不伪造测试结果
  8. 不提交真实 API Key
  9. 不新增数据库服务
  10. 不把 mock web search 宣传为真实联网

测试验证计划

完成后运行以下命令验证构建和代码质量：

npm run lint
npm run typecheck
npm run build
npm run check
npm test

如果支持 RAG 命令，运行：

npm run rag:ingest -- --subject physics_mechanics
npm run rag:query -- --subject physics_mechanics \
  --question "非零初始高度斜抛运动如何求落地时间和射程？"

08 · Appendix

附录

A. 核心技术链

知识库 → RAG → Agent 质检 → 结构化教学输出 → Deep Interaction → Artifact → 教学资产库

这条链路是 STEMotion 的技术主线。知识库提供可信知识来源，RAG 负责检索与组装，Agent 质检确保输出质量，结构化教学输出面向学生和教师两个角色，Deep Interaction 将抽象知识转化为可交互的实验 artifact，最终沉淀到教学资产库供复用。

D. 术语表

RAG：Retrieval-Augmented Generation，检索增强生成，通过检索外部知识库来增强大语言模型的输出质量。

Deep Interaction：STEMotion 的交互实验生成管线，通过 Guided Planning、多 Agent 评审和 Repair 机制生成高质量的 HTML/SVG/Canvas 交互 artifact。

LearningBlueprint：学习蓝图，描述交互实验的教学目标、交互方式和评估标准的结构化文档。

Guided Planning：引导式规划，Deep Interaction 管线的第一步，将用户 prompt 转化为结构化的实验设计方案。

Artifact：教学资产的载体，可以是交互实验 HTML、学习报告、备课方案等可复用的教学内容。
